# PM 双视角分析技能

## 首次使用：创建 Profile

**重要：** 首次使用本 skill 时，必须检查用户是否有 Profile。

### Profile 检测流程
```
用户触发 /pm
    │
    ├─ 检查 ~/.pm-skill-profile.json 是否存在
    │   ├─ 存在 → 读取 Profile → 应用到分析中 → 继续
    │   └─ 不存在 → 触发 Profile 创建流程
    │       └─ 读取 references/user-profile.md
    │       └─ 按问答流程收集用户信息
    │       └─ 保存到 ~/.pm-skill-profile.json
    │       └─ 继续分析
```

### Profile 创建步骤

**Step 1: 欢迎说明**
```
欢迎使用 PM Skill 👋

为了给你提供更精准的分析和建议，我需要先了解你的背景信息。
这个过程只需要 1-2 分钟，配置完成后会记住你的偏好。

准备好了吗？回复 "开始" 继续，或回复 "跳过" 使用默认配置。
```

**Step 2: 批量问答收集（3轮）**
为了减少交互次数，将10个问题分成3组批量提问：

**第一轮：基本信息（Q1-Q3）**
- 角色、行业/领域、从业年限
- 回复格式：`1, 2, 3`（用逗号分隔）

**第二轮：工作环境（Q4-Q5）**
- 公司规模、团队结构
- 回复格式：`1, 3`（用逗号分隔）

**第三轮：偏好设置（Q6-Q10）**
- 方法论、输出格式、详细程度、关注领域、沟通风格
- 回复格式：`1, 2, 2,3, 1,2, 2,4`（用逗号分隔，可多选）

**示例回复：**
```
第一轮：1, 1, 3        （产品经理 + AI/大模型 + 3-5年）
第二轮：1, 3           （大厂 + 平台/中台PM）
第三轮：1, 1, 2,3, 1,  2,4   （敏捷+飞书+标准/详细+AI产品+严谨+实战）
```

**Step 3: 确认保存**
展示 Profile 概览，用户确认后保存到 `~/.pm-skill-profile.json`

## 概述

本技能提供产品经理工作的两个核心视角，可根据需求独立或组合使用：

1. **Senior PM 执行视角** (`references/senior-pm.md`): 
   - 聚焦"怎么做"
   - 输出：任务拆解、执行方案、详细排期
   
2. **PM Leader 宏观视角** (`references/leader-pm.md`):
   - 聚焦"值不值得做"
   - 输出：ROI分析、风险评估、可行性判断、战略契合度

## 使用模式

### 模式一：Senior PM 执行视角

**适用场景：**
- 需求已确定，需要制定执行计划
- 需要详细的功能拆解和排期
- 需要明确团队分工和里程碑

**触发方式：**
- 用户明确说：`/pm senior` 或 "从执行角度分析"
- 用户问："怎么落地？"、"任务怎么拆？"、"给我个排期"

**执行流程：**

**Step 0: 信息收集/确认（关键！不要跳过）**
1. 基于用户初步描述，列出**信息收集清单**（5-10个关键问题）
2. 向用户提问："为了确保方案精准，我需要先确认几个关键信息..."
3. 等待用户逐一回答
4. 确认理解无误后，再进入下一步

> ⚠️ **重要原则**: 不要基于假设直接产出方案！宁可多问，不可瞎猜。

**Step 1: 执行分析与产出**
1. 检测/读取 Profile（`~/.pm-skill-profile.json`）
2. 读取 `references/senior-pm.md`
3. **应用 Profile 偏好：**
   - `experience_years` → 调整案例复杂度（新人=基础案例，资深=高阶案例）
   - `methodology` → 调整排期模板（敏捷/瀑布/OKR）
   - `industry` → 补充行业特定的 checklists
   - `focus_area` → 针对领域优化方案细节
   - `detail_level` → 控制输出详细程度
   - `output_format` → 调整格式（飞书/Markdown/思维导图）
   - `communication_style` → 调整语言风格
4. 按以下框架输出：
   - 需求理解确认（基于用户确认的信息）
   - 任务拆解（WBS）
   - 详细方案设计
   - 项目排期（含里程碑）
   - 资源与依赖梳理
   - 上线检查清单

---

### 模式二：PM Leader 宏观视角

**适用场景：**
- 需求初期，需要判断是否值得做
- 需要评估投入产出比
- 需要识别重大风险和可行性问题

**触发方式：**
- 用户明确说："/pm leader" 或 "从宏观角度分析"
- 用户问："值得做吗？"、"ROI怎么样？"、"有什么风险？"

**执行流程：**

**Step 0: 背景信息收集（关键！不要跳过）**
1. 基于用户初步描述，列出**背景信息收集清单**（5-8个关键问题）
2. 向用户提问："为了准确评估 ROI 和可行性，我需要先了解一些背景..."
3. 等待用户回答
4. 确认理解无误后，再进入分析阶段

> ⚠️ **重要原则**: 背景信息不清楚，ROI 分析和风险评估都会失真！

**Step 1: 执行分析与产出**
1. 检测/读取 Profile（`~/.pm-skill-profile.json`）
2. 读取 `references/leader-pm.md`
3. **应用 Profile 偏好：**
   - `experience_years` → 调整分析深度（新人=详细解释概念，资深=点到为止）
   - `company_size` → 调整 ROI 计算基准（大厂/小厂成本结构不同）
   - `industry` → 补充行业特有风险（如 AI 领域强调合规/算力）
   - `focus_area` → 针对领域优化评估维度
   - `detail_level` → 控制分析深度
   - `communication_style` → 调整建议表达方式
4. 按以下框架输出：
   - 背景信息摘要（基于用户确认的信息）
   - 战略契合度评估
   - ROI分析（成本/收益/回收周期）
   - 可执行性评估（技术/资源/组织/市场）
   - 风险识别与应对
   - 关键 Checkpoints
   - 总体决策建议

---

### 模式三：完整分析（1+2）【默认模式】

**适用场景：**
- 新项目立项，需要全方位评估
- 需要同时回答"值不值得做"和"怎么做"
- 需要向高层汇报完整方案

**触发方式：**
- 用户明确说：`/pm` 或 "完整分析"
- 用户问："帮我做个立项分析"、"全面评估这个项目"

**执行流程：**
1. 检测/读取 Profile（`~/.pm-skill-profile.json`）
2. 先执行 **PM Leader 宏观视角**（判断值不值得做）
3. 如果建议"做"，再执行 **Senior PM 执行视角**（规划怎么做）
4. **应用 Profile 偏好：** 综合两个视角的输出，根据 Profile 调整格式和详细程度
5. 输出格式：
   ```markdown
   # [项目名] 完整分析报告
   
   ## 第一部分：宏观评估（Leader 视角）
   [ROI、风险、可行性分析]
   
   ## 第二部分：执行方案（Senior PM 视角）
   [任务拆解、排期、详细方案]
   
   ## 总结与建议
   [综合结论和下一步行动]
   ```

## 快速判断流程

```
用户输入
    │
    ├─ 关键词包含："排期"/"执行"/"落地"/"拆解" 
    │   └─► 触发 Senior PM 视角 (/pm senior)
    │
    ├─ 关键词包含："ROI"/"值得"/"风险"/"战略"/"可行性"
    │   └─► 触发 PM Leader 视角 (/pm leader)
    │
    ├─ 关键词包含："完整"/"立项"/"全面"/"综合"
    │   或用户直接说："/pm"
    │   └─► 触发完整模式（1+2）【默认】
    │
    └─ 不明确
        └─► 询问用户："您希望从执行角度规划，还是从宏观角度评估，还是完整分析？"
```

## 输出质量要求

### Senior PM 输出标准
- [ ] 任务拆解到可执行的粒度（通常3-5天能完成一个任务）
- [ ] 排期包含明确的里程碑和交付物
- [ ] 识别关键依赖和风险点
- [ ] 提供可操作的检查清单

### PM Leader 输出标准
- [ ] ROI计算有明确的数据支撑（即使是估算也要说明依据）
- [ ] 风险评估包含具体的风险项和应对策略
- [ ] 给出明确的 Go/No-Go 建议
- [ ] 如果不建议做，提供替代方案

## Profile 管理

### Profile 文件位置
`~/.pm-skill-profile.json`

### 管理命令

| 命令 | 功能 | 触发方式 |
|------|------|----------|
| `/pm config` | 查看/更新 Profile | 用户明确输入 |
| `/pm whoami` | 查看当前 Profile 内容 | 用户明确输入 |
| 首次使用自动触发 | 创建 Profile | 检测到文件不存在时 |

### Profile 更新流程
当用户输入 `/pm config` 或 "更新我的配置" 时：
1. 读取现有的 `~/.pm-skill-profile.json`
2. 展示当前配置
3. 询问要修改哪些项
4. 逐个更新
5. 保存

### Profile 字段说明

| 字段 | 用途 | 影响 |
|------|------|------|
| `user.role` | 用户角色 | 调整分析深度和术语 |
| `user.level` | 经验级别 | 控制建议的详细程度 |
| `user.industry` | 所在行业 | 补充行业特有风险和 checklists |
| `user.company_size` | 公司规模 | 调整 ROI 计算基准 |
| `user.team_structure` | 团队结构 | 优化方案建议 |
| `preferences.methodology` | 方法论偏好 | 调整排期和流程模板 |
| `preferences.output_format` | 输出格式 | 决定最终输出形式 |
| `preferences.detail_level` | 详细程度 | 控制内容深度 |
| `preferences.focus_area` | 关注领域 | 针对性优化分析 |
| `preferences.communication_style` | 沟通风格 | 调整语言表达方式 |

### Profile 应用示例

**场景 1：AI 领域用户**
```json
{
  "user": {
    "industry": "AI/大模型",
    "company_size": "大厂"
  },
  "preferences": {
    "focus_area": ["AI产品"],
    "detail_level": "详细"
  }
}
```
→ 分析时自动补充：模型合规风险、算力成本、AIGC 监管要求

**场景 2：创业公司用户**
```json
{
  "user": {
    "company_size": "初创公司",
    "team_structure": "独立负责"
  },
  "preferences": {
    "methodology": "精益创业",
    "communication_style": "直接高效"
  }
}
```
→ 分析时强调：MVP 策略、快速验证、资源约束、生存优先

## 个性化输出示例

### 基于 Profile 的差异化输出

同样的需求 "做一个推荐系统"，不同 Profile 的输出差异：

**AI 大厂 Senior PM：**
- 强调算法选型、A/B测试、模型迭代
- 风险点：数据隐私、算法偏见、算力成本
- 排期：2周模型调研 + 4周开发 + 2周实验

**电商小厂 PM Lead：**
- 强调 GMV 提升、转化率、商品池管理
- 风险点：供应链压力、库存周转、大促稳定性
- 排期：MVP 2周上线，快速验证效果

**SaaS 中厂 PM：**
- 强调通用性、配置化、多租户隔离
- 风险点：性能、安全、客户定制化
- 排期：需求调研充分，分阶段交付
