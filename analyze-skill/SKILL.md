# /analyze — AI Evaluation Report

把一份多模型评测飞书表格转为**固定样式**的中文 Markdown 报告（EU8-hu 样式）。

---

## 触发

- 用户输入 `/analyze <URL>`，或含有飞书 sheet/wiki 链接 + 评测/汇总/GSB 类关键词
- URL 形态：`.../wiki/<wiki_token>?sheet=<sheet_id>` 或 `.../sheets/<token>?sheet=<sheet_id>`
- 可选参数（用户自然语言说明）：目标语种过滤（如"只看 hu 语种"）

## 依赖

- `prod:lark-sheet-skill`（`sheet-meta` + `read-cells`）
- Python 3 标准库（analyze.py 在技能目录内）

---

## 工作流（5 步）

### Step 1. 解析 URL 与列结构

1. 从 URL 提取 `wiki_token/sheet_token` 与 `sheet_id`
2. 调 `sheet-meta` 确认 sheet 存在
3. 调 `read-cells <token> <sheet_id>!A1:<end>` 拉全量数据 → `raw.json`
4. 读首行，**交互/自动**识别以下列位置：
   - GSB 列（常名：GSB / winner）
   - Remarks 列（常名：Comment / Remarks / 备注）
   - 文件名列（含 `_<src>_<tgt>.json` 结构，用于解析语种）
   - 模型指标块：每个模型紧跟若干 0/1 列（BGM/sync/voiceover/audio/voiceover_consistent/gender/language…）+ 若干 1-5 列（translation/info/timbre…）

### Step 2. 过滤有效样本

- 只保留 GSB 列非空的行
- 若用户指定语种过滤，按文件名解析出的 target_lang 过滤子集

### Step 3. 核心指标口径（严格遵守）

| 指标 | 公式 |
|---|---|
| **GSB 分布** | winner 列归一化后计数 / 有效样本 N |
| **得分率** | (Σ(0/1 × 5) + Σ(1-5 原值)) / (已填槽位 × 5) |
| **可用率（槽位）** | 可用槽位 / 已填槽位；0/1=1 或 1-5≥4 记可用；**分母 = 各维度已填数量之和，不按 N×维度数估算** |
| 单项通过率 | 0/1=1 的比例 |
| 单项可用率 | 1-5≥4 的比例 |
| 单项均分 | 1-5 平均值 |

### Step 4. 文本挖掘（Remarks 句级归因）

- 按 `[.!。]\s+` 切句
- 每句识别提及的模型（关键字匹配 + 可配置 `keywords`）
- 每句命中缺陷词典分类（机械感 / 语速 / 音量 / 语调 / 口音 / 音色切换 / 语种/方言错误 / 噪声 / 同步 / 翻译 / 情绪）
- 缺陷计入该句提及的模型

### Step 5. 生成 EU8-hu 样式报告

严格按下方模板输出。

---

## 输出模板（固定样式 — EU8-hu 打样）

报告必须包含且仅包含以下 5 个章节，章节编号与表格结构不得修改：

```
# <场景名> — <模型A> vs <模型B> 测评汇总

**数据范围**：<sheet 路径> 中 <过滤条件>｜共 **<N_all>** 条，**<N_valid>** 条已填 GSB｜源语：<src 分布>｜目标语种：<tgt 分布>

> ⚠️ 数据完整度提醒：<如有覆盖率 <70% 的维度，逐条列出>；否则省略此段

---

## 1. GSB 结果

| 结果 | 数量 | 占比 |
|---|---:|---:|
| **<winner1>** | N | XX.X% |
| <winner2> | N | XX.X% |
| Same Good（并列好） | N | XX.X% |
| Same Bad（并列差） | N | XX.X% |

> <一句话总结：净胜率 / 哪个模型胜出 / 与其它语种对比>

---

## 2. 得分率（加权总分）

口径：0/1 维度 ×5，1-5 维度原值；按已填槽位计满分。

| 模型 | 实际得分 | 满分 | **得分率** |
|---|---:|---:|---:|
| **<模型A>** | X | X | **XX.XX%** |
| <模型B> | X | X | **XX.XX%** |

---

## 3. 可用率（按槽位计）

口径：0/1 取值=1 记可用，1-5 取值≥4 记可用；分母 = 各维度已填总槽位之和。

| 模型 | 可用槽位 | 总槽位 | **可用率** |
|---|---:|---:|---:|
| **<模型A>** | X | X | **XX.XX%** |
| <模型B> | X | X | **XX.XX%** |

---

## 4. 单项指标拆解

| 维度 | 类型 | <模型A> | <模型B> | 差值 |
|---|:-:|---:|---:|---:|
| bgm_complete | 0/1 | XX.X% (a/n) | XX.X% (a/n) | +/-X.X |
| ... | ... | ... | ... | ... |
| <1-5 项> | 1-5 ≥4 | XX.X% (均分 X.XX) | XX.X% (均分 X.XX) | +/-X.X |

> **规则**：
> - 差值 = 模型A − 模型B（绝对百分点）；差值 ≥10pt 的行，左右两侧数值用粗体
> - 1-5 项必须同时显示"≥4 率"与"均分"
> - 模型A 默认为 GSB 胜出的模型；如并列则按 skill 内部 config 顺序

---

## 5. 关键洞察

<此处由 LLM 基于前 4 节数据生成 3–5 条分析式 bullet，必须满足：>
1. **第 1 条**：定位决胜维度（差值最大的 1–2 个指标），给出具体百分点差距
2. **第 2 条**：指出共性问题（两模型都低于 70% 的维度），若无则省略
3. **第 3 条**：用 Remarks 文本挖掘结果佐证量化结论（引 1–2 个高频缺陷词 + 次数）
4. **最后一条**：给出选型建议，指明弱势模型的核心改进方向

**风格要求**：
- 不要机械罗列数字，要有"因为 X，所以 Y"的因果链
- 数据可信度存疑时（如某维度覆盖 <70%），必须在洞察里点出并弱化其权重
- 结论要直接可执行（"hu 语种应优先采用 voice_clone"，而不是"建议综合考虑"）
```

---

## 格式硬约束

1. 全中文标题；无 emoji（除 ⚠️ 用于数据完整度告警）
2. 百分比统一 **1 位小数**（得分率/可用率 2 位小数，用于区分）
3. 表格数值列用 `---:` 右对齐
4. **禁止**在正文用 code block 包裹表格
5. 所有量化信息必须用表格；段落只用于"数据范围"一行和"关键洞察"列表
6. 缺陷文本挖掘未触发时，允许省略 §5 中第 3 条
7. 模型名保留英文原名（`voice_clone`、`voice_id`、`Heygen`、`v2`…）

---

## 输出后行为约束（重要）

**分析完成后，直接输出报告，禁止主动提问或追问用户。**

❌ **禁止出现的内容：**
- "需要按单一目标语种切片做分语种小报告吗？"
- "需要导出至飞书文档/多维表格沉淀基线吗？"
- "是否需要进一步分析...？"
- "您还需要...吗？"
- 任何以"需要"、"是否"、"您还想"等开头的追问句式

✅ **正确做法：**
- 报告输出完毕即结束
- 如果用户有后续需求，会主动发起新的指令
- 保持沉默，等待用户下一步指令

## 脚本入口

```bash
# 1. 拉数据
python3.11 -m lark_sheet read-cells <token> "<sheet>!A1:<end>" --value-render ToString > raw.json
sed -i '1d' raw.json

# 2. 写 config（列映射 + 模型别名）
# 见 examples/config_eu8.json

# 3. 出报告
python3 analyze.py --raw raw.json --config config.json --out report.md
```

`analyze.py` 产出包含前 4 节数据的结构化 Markdown；**§5 关键洞察**由 LLM 基于前 4 节数据生成（脚本不强行模板化，保留分析质量）。
